# Design Ars Festival UI

## Start with the reference

Read [references/ui-ux-principles.md](references/ui-ux-principles.md) completely before making visual or UX decisions. Treat it as a design interpretation of the official 2026 festival site, not as a legal brand manual.

## Follow this design process

1. Clarify the page's audience, primary task, content, language needs, and relationship to the main festival.
2. Select the closest page pattern from the reference: landing, editorial/theme, program, location, or event detail.
3. Establish the two-layer identity: a restrained Ars Electronica frame plus an expressive sub-event layer.
4. Define the information hierarchy before styling. Make purpose, date, place, and next action immediately understandable.
5. Apply the color, typography, composition, imagery, navigation, and interaction principles from the reference.
6. Adapt the composition deliberately for narrow screens; do not merely shrink the desktop arrangement.
7. Review the concept against the accessibility and quality checklist in the reference.

## Make design decisions, not code decisions

Describe intent in terms of hierarchy, rhythm, emphasis, behavior, and user outcome. Do not prescribe frameworks, component libraries, CSS, tokens, class names, breakpoints, or build tooling unless the user explicitly asks for implementation after the design direction is settled.

When an existing product or repository already defines a shell, preserve it and apply this skill to the event-specific experience inside that shell.

## Handle brand tension consciously

Prefer recognizable kinship over literal duplication. Preserve the official site's character while allowing the sub-event to have one clear concept of its own.

If visual fidelity conflicts with comprehension, accessibility, or the sub-event's primary task, keep the festival character but improve the experience. Never rely on color alone, and do not reproduce marginal contrast simply because it appears on the reference site.

## Expected outputs

Produce the artifact requested by the user, such as:

- a UI/UX direction or design brief;
- information architecture and navigation rationale;
- page-by-page layout guidance;
- a wireframe or mockup specification;
- a critique of an existing design;
- design acceptance criteria for later implementation.

Explain the few most consequential choices. Keep the output focused on design and user experience.
